# Опросы в мессенджерах

Цель: собрать добровольные отзывы о компании из Telegram, Max и VK через ботов, которых представитель ПлюсЗвена добавляет в профильные группы или чаты.

Бот не должен читать историю чата и сохранять групповые сообщения как отзывы. В группе он публикует короткий запрос и просит писать в личные сообщения. В отчет попадают только ответы из лички после модерации.

## Поток

1. Представитель собирает первичный отчет по ИНН.
2. Создает кампанию опроса:

```powershell
$body = @{
  inn = "7709787790"
  company_name = 'ООО "ВЕЛЕССТРОЙ"'
  platforms = @("telegram", "max", "vk")
  chat_ids = @("-1001234567890")
} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8788/api/messenger-campaigns" -ContentType "application/json; charset=utf-8" -Body $body
```

3. Бот публикует в группе текст кампании.
4. Участник пишет боту в личку код кампании и факты.
5. Адаптер бота отправляет нормализованный ответ в `/api/messenger-inbox`.
6. Ответ сохраняется в `data/company_registry.json` и появляется в отчете как источник `messenger_reviews`.

## Telegram MVP

Создать бота через BotFather, добавить его в нужные группы, выдать право писать сообщения.

```powershell
$env:TELEGRAM_BOT_TOKEN = "token_from_botfather"
$env:MESSENGER_INBOX_SECRET = "local-long-random-secret"
powershell -ExecutionPolicy Bypass -File .\telegram_review_bot.ps1 `
  -Inn 7709787790 `
  -CompanyName 'ООО "ВЕЛЕССТРОЙ"' `
  -ChatIds "-1001234567890" `
  -PostPrompt
```

Скрипт:

- создает кампанию, если `-CampaignId` не передан;
- отправляет текст запроса в `-ChatIds`, если указан `-PostPrompt`;
- читает только личные сообщения боту через long polling;
- сохраняет ответы в локальный `/api/messenger-inbox`;
- не сохраняет Telegram user id напрямую, только хэш.

## VK Callback MVP

Создать сообщество VK, включить сообщения сообщества и Callback API. Для локальной разработки нужен публичный HTTPS-туннель до `http://127.0.0.1:8791/`.

```powershell
$env:VK_CALLBACK_CONFIRMATION = "confirmation_string_from_vk"
$env:VK_CALLBACK_SECRET = "secret_from_vk_callback_settings"
$env:VK_GROUP_TOKEN = "group_access_token"
$env:MESSENGER_INBOX_SECRET = "local-long-random-secret"
powershell -ExecutionPolicy Bypass -File .\vk_review_bot.ps1 `
  -Inn 7709787790 `
  -CampaignId "pz-7709787790-20260705120000" `
  -CompanyName 'ООО "ВЕЛЕССТРОЙ"'
```

Скрипт:

- возвращает `VK_CALLBACK_CONFIRMATION` на событие `confirmation`;
- проверяет `secret`, если задан `VK_CALLBACK_SECRET`;
- принимает `message_new`;
- сохраняет текст как ответ платформы `vk`;
- при наличии `VK_GROUP_TOKEN` отвечает пользователю в VK.

## Max Bridge MVP

Для Max оставлен bridge с единым входящим форматом, потому что формат webhook/update нужно сверить с фактическим bot API аккаунта. Bridge принимает POST на `http://127.0.0.1:8792/`, достает текст, автора и чат из распространенных полей и отправляет ответ в `/api/messenger-inbox`.

```powershell
$env:MAX_BRIDGE_SECRET = "secret-for-public-adapter"
$env:MESSENGER_INBOX_SECRET = "local-long-random-secret"
powershell -ExecutionPolicy Bypass -File .\max_review_bridge.ps1 `
  -Inn 7709787790 `
  -CampaignId "pz-7709787790-20260705120000" `
  -CompanyName 'ООО "ВЕЛЕССТРОЙ"'
```

Внешний Max-адаптер или webhook должен передать заголовок:

```text
X-Max-Bridge-Secret: secret-for-public-adapter
```

## Единый входящий формат

Max и VK должны приводить входящие личные сообщения к этому JSON:

```json
{
  "inn": "7709787790",
  "company_name": "ООО \"ВЕЛЕССТРОЙ\"",
  "campaign_id": "pz-7709787790-20260705120000",
  "platform": "telegram",
  "chat_id": "source-chat-or-dialog-id",
  "author_id": "platform-user-id",
  "private_message": true,
  "consent": true,
  "role": "сотрудник",
  "period": "2024",
  "object_or_city": "Ямал",
  "text": "Работал монтажником. Зарплату задержали на две недели, проживание было как обещали.",
  "source_url": ""
}
```

`platform`: `telegram`, `max`, `vk` или `manual`.

`private_message` должен быть `true`; групповые сообщения не принимаются.

Если задана переменная окружения `MESSENGER_INBOX_SECRET`, все адаптеры должны передавать секрет в заголовке:

```text
X-PlusZveno-Secret: local-long-random-secret
```

## Модерация

Все входящие ответы получают статус `needs_moderation`. Перед публикацией нужно проверить:

- есть ли роль автора: сотрудник, подрядчик, заказчик, клиент;
- указан ли период и город/объект;
- нет ли персональных данных третьих лиц;
- не дублирует ли ответ уже сохраненный отзыв;
- не выглядит ли текст заказным или манипулятивным;
- относится ли отзыв именно к проверяемому юрлицу, бренду или объекту.

## Отложено: бот как дежурный проверяющий в чате

Если бота легально добавили в группу или чат и дали права читать новые сообщения, позже можно расширить его роль:

- читать только новые сообщения, доступные боту после добавления в чат;
- искать ИНН, ОГРН, названия компаний и ссылки на вакансии;
- запускать проверку компании или вакансии по найденному идентификатору;
- отвечать в чат короткой публичной карточкой: найдено юрлицо, статус проверки, есть ли сигналы, когда собран отчет;
- давать ссылку на закрытый отчет;
- открывать полный отчет только после авторизации и подтверждения роли/документов;
- просить отзывы и факты писать в личные сообщения боту, а не в общий чат.

В чате бот не должен молча сохранять весь поток сообщений. Для журнала проверки достаточно хранить:

- платформу и хэш чата;
- id или хэш сообщения;
- найденный ИНН/ОГРН/ссылку;
- хэш автора;
- время события;
- результат короткой проверки;
- ссылку на закрытую карточку отчета.

Разделение сценариев:

- чат: обнаружение компании/вакансии и короткий ответ со ссылкой;
- личка: добровольный отзыв или уточнение фактов;
- личный кабинет: полный отчет, авторизация, документы и подтверждение роли.

## Что еще нужно для продакшена

- отдельный секрет для `/api/messenger-inbox`, чтобы endpoint не принимал ответы от кого угодно;
- webhook-режим вместо long polling;
- отдельные адаптеры Max и VK;
- экран модерации ответов;
- классификация рисков: задержка оплаты, неоплаченная работа, безопасность, жилье, документы, давление руководства, дорога.
